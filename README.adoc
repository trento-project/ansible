= Trento Ansible
ifndef::site-gen-antora[:relfileprefix: docs/]

Collection of playbooks that aim in installing Trento and its
third-party dependencies.

== Support matrix

The playbooks can only be run on target nodes running SLES4SAP.

Refer to the following table for supported control and target node
combinations. The specified value in the target cell is the _Python_
interpreter version that supports the execution of the playbooks:

[cols=6*^]
|===
| Control node | SLES15-SP4 | SLES15-SP5 | SLES15-SP6 | SLES15-SP7 | SLES16

| ansible9 | python3(3.6)-only | python3(3.6)-only | python3(3.6)/3.11 | python3(3.6)/3.11 | X

| ansible11 | X | X | python3.11-only | python3.11-only | python3(3.13)-only
|===

NOTE: On SLES 15 SP5 and below, Ansible may default to the newest
Python interpreter, causing incompatibility issues between the
playbook and the target node. To prevent this, you must explicitly set
the interpreter in `vars.json`: to `"ansible_python_interpreter":
"/usr/bin/python3"`

== Components

Trento components:

* https://github.com/trento-project/web[web]
* https://github.com/trento-project/wanda[wanda]
* https://github.com/trento-project/checks[checks]
* https://github.com/trento-project/agent[agent]
* https://github.com/trento-project/mcp-server[mcp]

Third-party dependencies:

* postgresql
* rabbitmq
* prometheus
* nginx

The third parties are installed using `zypper` packages and configured
with dedicated roles.

The nginx configuration acts as a reverse proxy for all the components.


== Playbooks overview

=== Server

`server.yml` playbook installs Trento server components (Web and
Wanda) along with the supporting third-party application dependencies.
It can also install `supportutils` together with the
`supportutils-plugin-trento` package to make `supportconfig` collect
Trento-specific data on Trento server hosts. This can be disabled by
setting `supportutils_enabled` to `false` in the inventory.

This playbook supports auto-discovery of the hosts for RabbitMQ, Postgres and Prometheus
in Web and Wanda. This feature is enabled when `provision_postgres`,
`provision_rabbitmq` and `provision_prometheus` flags are true. When
auto-discovery is active, `*_host` variables (in `trento` role
and any other role using them) are ignored.
If you want to use external services for these dependencies,
turn off the respective flag and specify the `*_host` variable in your
inventory.

=== MCP

`mcp.yml` playbook installs only the Trento MCP Server. When the
`mcp_hosts` inventory group exists, the MCP Server is deployed there.
Otherwise, it is deployed on `trento_server`.

The role auto-selects how MCP discovers the Trento APIs based on
inventory layout. When the MCP Server runs on a `trento_server` host
(the default colocated case), it uses `OAS_PATH` against the local
Trento Web and Wanda backends on `127.0.0.1` — the most direct path,
with no DNS or TLS dependency. When the MCP Server runs on a dedicated
`mcp_hosts` machine, it uses `TRENTO_URL` pointing at
`https://<trento_server_name>` so the request flows through the
reverse proxy. The user does not configure the mode; it follows from
where the MCP Server is deployed.

The default and simplest deployment is to omit `mcp_hosts` and run the
MCP Server on `trento_server`. Use `mcp_hosts` for advanced deployments
where the MCP Server must run on a dedicated host. Dedicated MCP
deployments default `mcp_server_trento_url` to
`https://<trento_server_name>`; override it when Trento is reachable
through a different URL, for example without the reverse proxy or TLS.
When the dedicated MCP host cannot trust a self-signed proxy
certificate, set `mcp_server_insecure_skip_tls_verify: true` or install
the CA certificate in the system trust store outside this role.

Dedicated MCP host example:

[source,yaml]
----
all:
  children:
    mcp_hosts:
      hosts:
        trento-mcp:
          ansible_host: "your-mcp-host"
          ansible_user: "your-user"
          ansible_ssh_private_key_file: "/home/user/.ssh/id_rsa"
----

When `site.yml` is used, MCP is provisioned by importing `mcp.yml`
after `server.yml` by default. Set `provision_mcp` to `false` to skip
MCP provisioning from `site.yml`.

If `provision_proxy` is `true`, the endpoint is exposed through NGINX
at `https://<trento_server_name><trento_mcp_proxy_location>/mcp`.
When `mcp_server_transport` is set to `sse`, clients must use the same
base URL with the `/sse` path instead of `/mcp`.

If `provision_proxy` is `false`, or the MCP Server is deployed on a
dedicated `mcp_hosts` machine, the playbook opens
`trento_mcp_listen_port` in `firewalld` so the MCP service can be
reached directly on the systemd listener port. When
`mcp_server_health_enabled` is `true`, `trento_mcp_health_port` is
opened as well.

The Trento MCP Server does not install or configure an LLM client.
Generate a Personal Access Token in Trento Web and configure it in the
MCP client. With the default `mcp_server_header_name`, clients pass the
token through the `Authorization` header. The MCP endpoint exposed by
this playbook is:

  https://<trento_server_name>/mcp-server-trento/mcp

(or `/sse` instead of `/mcp` when `mcp_server_transport` is `sse`).

For up-to-date client configuration examples — including supported
clients and the exact JSON/YAML to set up the Trento PAT — refer to
the official Trento MCP Server usage guide:

https://www.trento-project.io/docs/user-guide/trento-mcp-how-to-use.html

For the full server configuration reference, see:

https://www.trento-project.io/docs/user-guide/trento-mcp-config.html

=== Agent

`agent.yml` playbook installs and configures Trento agent.
It can also install `supportutils` together with the
`supportutils-plugin-trento` package to make `supportconfig` collect
Trento agent diagnostics. This can be disabled by setting
`supportutils_enabled` to `false` in the inventory.

This playbook supports auto-discovery of the host for RabbitMQ for use
in agents. This feature is enabled when `provision_rabbitmq` flag is
true. When auto-discovery is active, `agent_rabbitmq_host` variables
in `agent` role is ignored. If you want to use external service for
this dependency, turn off the `provision_rabbitmq` flag and specify the
`agent_rabbitmq_host` variable in your inventory.

=== Site

`site.yml` is playbook that makes full Trento installation. It
installs both the server components and agents. Additionally, the API
key for the agents is automatically retrieved from Trento server and
passed on to the deployed agents. Thus, the value of
`agent_web_api_key` variables is ignored when using this playbook.
`site.yml` also provisions the Trento MCP Server through `mcp.yml`
after the server playbook completes. Set `provision_mcp` to `false` to
opt out.

== Usage

=== 1. Clone the repository

`git clone https://github.com/trento-project/ansible.git`

=== 2. Prepare your inventory file

Get to the `ansible` directory: `cd ansible`

Make sure all hosts with active roles allow access from the machine
that is executing the playbooks. If connecting to the machines with
SSH keys, you might want to execute:

[source, console]
----
$ ssh-copy-id <username>@<ip-of-the-host>
----

Create an `inventory.yml` file, defining the IP address or domain name
of the machines for each group expected in the playbook. You might use
the same machine in more than one group. The `mcp_hosts` group is
optional; if it is omitted, MCP falls back to `trento_server`.

Example:

[source,yaml]
----
all:
  children:
    trento_server:
      hosts:
        vitellone:
          ansible_host: "your-host"
          ansible_user: "your-user"
          ansible_ssh_private_key_file: "/home/user/.ssh/id_rsa"
    postgres_hosts:
      hosts:
        vitellone:
          ansible_host: "your-host"
          ansible_user: "your-user"
          ansible_password: "your-password"
    rabbitmq_hosts:
      hosts:
        vitellone:
          ansible_host: "your-host"
          ansible_user: "your-user"
          ansible_ssh_private_key_file: "/home/user/.ssh/id_rsa"
    prometheus_hosts:
      hosts:
        vitellone:
          ansible_host: "your-host"
          ansible_user: "your-user"
          ansible_ssh_private_key_file: "/home/user/.ssh/id_rsa"
    agents:
      hosts:
        vitellone:
          ansible_host: "your-host"
          ansible_user: "your-user"
          ansible_password: "your-password"
----

Here, a mixture of password and SSH keys authentication is used for
the hosts.

=== 3. Setup playbook variables

Add the desired variables to the inventory. Use either `vars:` section
in the inventory file itself or use host_vars/group_vars directories:

[source,yaml]
----
all:
  children:
    trento_server:
      hosts: ...
    postgres_hosts:
      hosts: ...
    rabbitmq_hosts:
      hosts: ...
    prometheus_hosts:
      hosts: ...
    agents:
      hosts: ...

  vars:
    trento_server_name": "trento-deployment.example.com",
    trento_web_admin_password: "adminpassword",
    trento_web_postgres_password: "postgres",
    trento_wanda_postgres_password: "postgres",
    trento_rabbitmq_password: "guest",
    rproxy_ssl_cert: "<paste your SSL certificate here in base64>",
    rproxy_ssl_key: "<paste your SSL certificate key here in base64>"

    # Prometheus authentication
    trento_prometheus_auth: "basic"
    trento_prometheus_auth_username: "prometheus"
    trento_prometheus_auth_password: "your-prometheus-password"
----

=== 4. Run the playbooks

Prior to running the playbooks, tell ansible to fetch the required
modules:

[source, console]
----
$ ansible-galaxy collection install -r requirements.yml
----

Run a playbook:

[source, console]
----
$ ansible-playbook -i path/to/inventory.yml <playbook-name.yml>
----

[NOTE]
====
You could also use `--extra-vars` argument to `ansible-playbook`
command to override every variable in the playbook. We advise against
its usage but if the need arises you can create a `vars.yaml` file
with your overrides and pass it as following:

[source, console]
----
$ ansible-playbook -i path/to/inventory.yml <playbook-name.yml> --extra-vars "@path/to/vars.yaml"
----

The `@` character in front of the `vars.yaml` tells `ansible-playbook`
that the variables will not be specified in-line but as an external
file instead.
====

=== With docker container

You can use the docker image `ghcr.io/trento-project/ansible`, to run
this playbook, the image contains the playbook files ready to be
provisioned. The docker image assumes you mount an `inventory` file
and an `extra-vars` file.

Mounting your ssh socket will enable you to access the remote machines
like in your local environment.

Assuming you have in the current folder a file called `inventory.yml`
and `extra-vars.yaml`

[source, console]
----
$ docker run \
   -e "SSH_AUTH_SOCK=/ssh-agent" \
   -v $(pwd)/inventory.yml:/playbook/inventory.yml \
   -v $(pwd)/extra-vars.yaml:/playbook/extra-vars.yaml \
   -v $SSH_AUTH_SOCK:/ssh-agent \
   ghcr.io/trento-project/ansible:rolling /playbook/inventory.yml /playbook/extra-vars.yaml
----

== Clean up

In order to clean up most of the applied changes and created resources,
the cleanup playbooks could be used. They use the same inventory and
variables file than the main playbook.

`cleanup.yml` removes Trento server-related resources. These are the
cleaned resources:

* Stops Web and Wanda services
* Stops Trento MCP Server service
* Postgresql database and users
* Reverse proxy vhost configuration file
* RabbitMQ vhost
* `supportutils-plugin-trento` package
* Reverts Firewall rules

If `mcp_hosts` is present, `cleanup.yml` also removes the Trento MCP
Server from those dedicated hosts.

Run the playbook with:

[source, console]
----
$ ansible-playbook -i inventory.yml cleanup.yml
----

`cleanup-agents.yml` removes Trento agent-related resources, including
the `supportutils-plugin-trento` package on agent hosts.

`cleanup-mcp.yml` removes only the Trento MCP Server. When the core
Trento server stays in place and `provision_proxy` is `true`, the
reverse proxy is re-rendered without the MCP endpoint.

*Disclaimer: The installed packages are not removed as most of the times
they are of general usage, and this could have impact in many other
services. An exception is `supportutils-plugin-trento`, which is
removed because it is Trento-specific. The base `supportutils` package
is left installed.*
